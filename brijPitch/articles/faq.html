    Resume Addenda

    Since Mike "picked up his lance" again in 2016, he has worked on a huge array of software technologies, some of which are listed below.

    1. JavaScript: The Dojo Toolkit

    The original idea was to build Brij as a dBaas (Database As A Service). Naturally, I built the earlier versions with a JavaScript frontEnd and Java backEnd. For the toolkit I chose Dojo, which was originally built by IBM and was then heavily used internally in all IBM Enterprise Applications. Subsequent iterations of Brij moved to a desktop client version to target Enterprise Users, and Dojo was replaced first with Java Swing and eventually with dotNet.
    However, I did a considerable amount of work with Brij using Dojo, you can see a screen here showing a hierarchical DataView which I built by hand (Dojo allows true OOP in JavaScript)

    2. AI and Machine Learning

    In one of the middle stages of the Brij development timeline (Java Backend) I felt the need to use AI technologies to provide an auto-learning NLP tagging module, so I picked up and learnt Mallet, an open-source Java AI tool used by the US Military.
    I built, trained and ran ML models to perform the tagging based on test datasets. Mallet was funded by a research grant and had not been updated since a while, so I took a deep dive into the source code and made substantial improvements including adding the ability to use N-grams in training the models - this was fairly cutting edge at the time, but the AI sphere moves so fast it is probably considered old hat today.

    3. Java JDK internals

    Some UI elements in Java Swing were incomplete when Oracle bought the language from Sun Microsystems. I needed the code for the Brij User Interface; so I had to delve into the JDK and complete the code. Fortunately the coders at Sun had left a very well-commented and legible codebase.
    At the time the Java language had just begun to offer Map-Reduce and Streams which became my favorite enthusiasm.

    4. Java JDK Functional idioms

    I regularly combed the web (as I continue to do today) for all the latest developement paradigms: any tool to make Brij better was an additional arrow in my quiver. I used cutting edge tech like Google's guice and javax.inject, and kept a constant eye on the pulse of the industry.
    As the Brij System API matured, I began to use more and more Functional Programming Abstractions within Java. Java is of course not a functional programming language but you can code functionally in Java using workarounds.
    I did this for so long that on taking a step back I discovered that the bulk of my work lately had not been working in Core Brij functionality but rather improving the JDK by adding functional abstractions to it! To illustrate, here is a snippet of the kind of work I was doing ->

    The mTpl (then named mTuple)

     Optional<List> squeeze(List<mTuple<T>> inL)
    List outL = new ArrayList<>()
    if (inL.size() < 1) 
        return Optional.empty()
     else 
        inL.stream()
            .map( t -> t.getR())
            .filter((Optional ot) -> ot.isPresent())
            .map((Optional t)   -> outL.add(t.get()))
        prnOut.accept("squeeze -> after proc, new size = " + outL.size())
        return Optional.of(outL)

 Optional<T> filter(Predicate<? super T> predicate) 
    Objects.requireNonNull(predicate)
    if (!hasR())
        return Optional.empty()
    else
        return predicate.test(right.get()) ? right : Optional.empty()

<U> mTuple<U> map(Function<? super T, ? extends U> failable) 
    //sets this mTuple to < ExceptionMsg, Result >
    Optional r = Optional.empty()
    Optional<String> l = Optional.empty()
    Objects.requireNonNull(failable)
        try
            r = Optional.of(failable.apply(right.get()))
         catch (Exception ex) 
            l = Optional.of(ex.getMessage())
            prnEx.accept("Exception in mTuple.map -> " , ex)
    return new mTuple(l,r)

<U> mTuple<U> flatMap(Function<? super T, mTuple<U>> mapper) 
    Objects.requireNonNull(mapper)
    Optional r = Optional.empty()
    Optional<String> l = Optional.empty()
    if (hasR()) 
        try
            r = Objects.requireNonNull(mapper.apply(right.get()).getR())
         catch (Exception ex) 
            l = Optional.of(ex.getMessage())
            prnEx.accept("Exception in mTuple.flatMap-> " , ex)
    return new mTuple(l,r)

<R> mTuple<R> lift(mTuple m1, mTuple m2, BiFunction<Object, Object, mTuple> lifter) 
    return m2.flatMap( second -> 
            m1.map ( first -> lifter.apply( first, second ) )
    )

    Anonymous

    Since it was not my priority to do language programming - other languages already had these abstractions - I decided to ditch java in favor of a purely functional programming language. I had already been reading and evaluating Scala, Clojure and other alternatives; and after a while I decided to port Brij to F#.

    5. Going Purely Functional

    I quickly became proficient in Functional Programming Paradigms, and was shocked and delighted to discover the rich tooling and endless possibilities in this new world; especially all the research work already been done in languages like Haskell. I became an F# enthusiast. Rewriting the Brij system in a functional language was hairy at times but I now had serious tools - including metaprogramming - to tackle the challenges I was facing.
    As before, I continued to research paradigms, techniques, patterns: anything I could use now or possibly use later. I became familiar enough in languages like Haskell and OCaml to port code from them to F#.
    With the move to F#, Brij went from possible to done.

    6. Writing compilers

    I knew of the existence of tools like Antler and had worked with Byte Code Manipulation tools like Lombok, byteBuddy and aspectJ to either manipulate code in memory or before/after compilation.
    The final tools which I picked up and used in my journey to bring Brij to market were FsLex/FsYacc and FParsec. These tools enabled me to compile raw F# code in order to translate Brij documents to MongoDB queries on the fly.

    10x Developer

    Tip

    Q: Can you provide further validation of your claim to be a 10x Developer?

    Here is one experience which opened my eyes:
    At about a year into a consulting gig at a major Philadelphia Finance firm (at their downtown HQ which was a 20+ floor building with about 3k employees) the company imploded. Within weeks the skyscraper became a ghost town; I was one of the last dozen people working there. Employees scrambled to find new positions (post Y2K) but as a consultant, I didn't care: I was paid by the hour and happy to work as long as they wanted me.
    Many key developers had left and the main function of the acting CTO was to give outside financial companies (which had bought pieces of this company) the data dumps they needed to transition the clients.
    My background uniquely qualified me to harvest the data and shape it per clients' needs. One manager - who I'd never worked with before - asked me for an estimate to code 20 Database Views for a database I had not seen before. I told her it would take a week and a half.
    "What do you mean, a week and a half? For how many?" she asked. I explained that my estimate was for 45 minutes Median per View, plus a 50% safety margin for expected difficulties; which amounted to 60 hours.
    "That's not what the other developer quoted," she said, referring to an absent colleague.
    "What did he quote?" I asked.
    "A week for each View," she replied.

    Solo Effort

    While I have worked as part of a team at many of my contracts, this has not been much of a choice for me: I usually work alone, even on gigs. I am a huge enthusiast of the many famous one-man-army developers - to me they resonate with the image of a lone cowboy.
    As my personal history shows, all my best work has occured when working solo (with or without a Manager). I honestly believe that I am one of the people who are most productive on their own: my personality traits include a (people have said remarkable) memory which allows me to focus on tiny aspects while retaining a macro view; and to clearly see the ramifications of any product change or improvement across all aspects of the product.
    Having taken many ideas from concept to market has refined my approach in terms of both knowing where to find and how to avoid bugs or bottlenecks and - on the other hand - knowing how to improve an aspect; which aspect to improve; and how each possible feature could enhance or detract from product value.
